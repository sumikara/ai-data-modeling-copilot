# SCD, Default Rows, Incremental Load, and Procedure Patterns

## Purpose
Define project-local SCD behavior, default-row governance, incremental-load constraints, and procedure templates for repeatable retail DWH loading.

## AI Copilot Usage Rules
The AI must:
- avoid finalizing SCD type without human approval when policy is ambiguous,
- propose SCD candidates with trade-offs,
- enforce employee SCD2 as a known project rule,
- keep default-row behavior explicit in design and SQL,
- verify that incremental procedures preserve entity behavior semantics.

## SCD Strategy

### Type 0
Insert-only immutable entities.

Used for:
- promotions as event versions,
- geography lookups,
- delivery profiles when treated as stable configuration,
- other lookup-like entities.

### Type 1
Overwrite current descriptive attributes.

Used when:
- only current descriptive state matters,
- historical tracking is not analytically required.

Possible entities:
- products,
- customers (depends on approved customer identity strategy).

### Type 2
History-preserving versioned entities.

Used for:
- employees.

## Employee SCD2

### Business Key
`employee_src_id`

### Surrogate Key
`employee_id` generated by sequence in 3NF.

### SCD2 Columns
- `start_dt`
- `end_dt`
- `is_active`
- `insert_dt`
- `update_dt`

### Logic Pattern
1. Stage latest employee candidates.
2. Compare staged row to active target row.
3. If descriptive attributes changed, expire active row.
4. Insert new active version.
5. Use default `end_dt = 9999-12-31`.
6. Log expired and inserted row counts.

### Hash Compare
Use MD5 across descriptive non-key attributes for change detection.

### `ROW_NUMBER()` vs `LEAD()`
`ROW_NUMBER()` should:
- remove exact duplicate versions,
- not collapse all history into one latest snapshot.

`LEAD()` should:
- derive `end_dt`,
- determine active/inactive status,
- close prior versions using next-version `start_dt`.

## DM Employee Matching
Fact-like loads should resolve employee dimension versions using this fallback order:
1. in-range match (`start_dt <= transaction_dt < end_dt`),
2. nearest prior version,
3. earliest available version,
4. default row (`-1`).

## Default Row Strategy
Every dimension should include an unknown/default row:
- surrogate key = `-1`,
- descriptive attributes = `'n.a.'` / `'N/A'`.

Purpose:
- keep fact rows loadable,
- preserve unresolved events,
- avoid silent row loss.

## Incremental Load Rules
- Incremental data must be provided in separate files.
- Foreign table file names should be generic and swappable.
- Do not hardcode timestamped file names in SQL/procedures.
- Replace incremental file content per run; do not append manually.
- SCD2 loads should insert new versions only when non-key attributes change.
- Procedures must avoid repeated inserts when no effective change occurred.

## Procedure Patterns

### SCD0 Pattern
- `INSERT` only,
- `WHERE NOT EXISTS`,
- `SELECT DISTINCT`,
- log inserted row counts.

### SCD1 Pattern
- `UPDATE` changed attributes,
- `INSERT` missing records,
- log updated and inserted counts.

### SCD2 Pattern
- expire changed active rows,
- insert new active versions,
- optional hash compare,
- log expired and inserted counts.

### Fact-Like Transaction Pattern
- insert-only,
- never update historical transaction events,
- use `row_sig` or composite-grain checks for duplicate protection.
